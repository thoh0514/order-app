# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 가능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드 : HTML, CSS, React, Javascript
- 백엔드 : Node.js, Express
- 데이터베이스 : PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결재 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 주문하기 화면 상세 요구사항

### 4.1 화면 개요
사용자가 커피 메뉴를 선택하고, 옵션을 커스터마이징한 후 장바구니에 담아 주문할 수 있는 메인 화면입니다.

### 4.2 화면 레이아웃

#### 4.2.1 헤더 영역 (상단)
- **브랜드명**: 좌측 상단에 "COZY" 브랜드명 표시
- **네비게이션 탭**: 우측 상단에 두 개의 탭 버튼
  - "주문하기" 탭: 현재 화면 (선택된 상태로 표시)
  - "관리자" 탭: 관리자 화면으로 이동하는 버튼

#### 4.2.2 상품 표시 영역 (중앙)
- **상품 그리드**: 가로로 여러 개의 상품 카드를 표시
- **상품 카드 구성 요소**:
  - 상품 이미지 (이미지가 없을 경우 플레이스홀더 표시)
  - 상품명 및 온도 옵션 (예: "아메리카노(ICE)", "아메리카노(HOT)", "카페라테")
  - 기본 가격 (원 단위로 표시, 예: "4,000원", "5,000원")
  - 상품 설명 (간단한 설명 텍스트)
  - 커스터마이징 옵션 (체크박스):
    - "샷 추가 (+500원)": 추가 샷 옵션 (추가 비용 500원)
    - "시럽 추가 (+0원)": 시럽 추가 옵션 (추가 비용 없음)
  - "담기" 버튼: 장바구니에 추가하는 버튼

#### 4.2.3 장바구니 영역 (하단)
- **장바구니 제목**: "장바구니" 라벨 표시
- **주문 목록**: 선택한 상품들의 목록 표시
  - 각 항목은 상품명과 선택한 옵션 정보 표시 (예: "아메리카노(ICE) (샷 추가)")
  - 각 항목 옆에 수량 표시 (예: "X 1", "X 2")
- **총 금액**: 우측에 "총 금액" 라벨과 함께 총 주문 금액 표시 (예: "총 금액 12,500원")
- **주문하기 버튼**: 장바구니 하단에 "주문하기" 버튼 배치

### 4.3 기능 요구사항

#### 4.3.1 상품 표시 기능
- 관리자가 등록한 모든 커피 메뉴를 화면에 표시
- 각 상품은 이미지, 이름, 가격, 설명 정보를 포함
- 상품은 그리드 형태로 배치하여 여러 개를 한 번에 볼 수 있도록 함

#### 4.3.2 옵션 선택 기능
- 각 상품마다 커스터마이징 옵션 제공
- 옵션은 체크박스 형태로 표시
- 옵션 선택 시 추가 비용이 있는 경우 가격 표시
- 여러 옵션을 동시에 선택 가능

#### 4.3.3 장바구니 기능
- "담기" 버튼 클릭 시 선택한 옵션과 함께 장바구니에 추가
- 같은 상품과 옵션 조합이면 수량 증가
- 장바구니에 담긴 상품 목록을 실시간으로 표시
- 각 상품의 옵션 정보를 명확히 표시
- 총 금액을 자동으로 계산하여 표시

#### 4.3.4 주문 기능
- "주문하기" 버튼 클릭 시 현재 장바구니의 모든 상품을 주문으로 전송
- 주문 완료 후 장바구니 초기화
- 주문 성공 시 사용자에게 피드백 제공 (알림 또는 메시지)

### 4.4 데이터 요구사항

#### 4.4.1 상품 데이터
- 상품 ID
- 상품명
- 온도 옵션 (ICE/HOT)
- 기본 가격
- 상품 설명
- 상품 이미지 URL (선택사항)

#### 4.4.2 옵션 데이터
- 옵션 ID
- 옵션명 (예: "샷 추가", "시럽 추가")
- 추가 가격

#### 4.4.3 장바구니 데이터
- 상품 ID
- 선택한 옵션 목록
- 수량
- 개별 항목 가격 (기본 가격 + 옵션 가격)
- 총 가격

#### 4.4.4 주문 데이터
- 주문 ID
- 주문 일시
- 주문 항목 목록 (상품, 옵션, 수량, 가격)
- 총 주문 금액
- 주문 상태 (대기/준비중/완료 등)

### 4.5 사용자 인터랙션

#### 4.5.1 상품 선택 플로우
1. 사용자가 원하는 상품 카드를 확인
2. 원하는 옵션을 체크박스로 선택
3. "담기" 버튼 클릭
4. 장바구니에 추가되고 장바구니 영역에 표시

#### 4.5.2 주문 플로우
1. 장바구니에 원하는 상품들을 모두 추가
2. 총 금액 확인
3. "주문하기" 버튼 클릭
4. 주문 완료 처리 및 장바구니 초기화

### 4.6 UI/UX 요구사항
- 반응형 디자인으로 다양한 화면 크기에 대응
- 직관적인 버튼 배치와 명확한 라벨링
- 장바구니에 추가된 항목은 즉시 시각적 피드백 제공
- 가격 정보는 명확하고 읽기 쉽게 표시
- 옵션 선택 상태를 명확히 표시

## 5. 관리자 화면 상세 요구사항

### 5.1 화면 개요
관리자가 주문 현황을 확인하고, 재고를 관리하며, 새로운 주문을 접수할 수 있는 관리자 대시보드 화면입니다.

### 5.2 화면 레이아웃

#### 5.2.1 헤더 영역 (상단)
- **브랜드명**: 좌측 상단에 "COZY" 브랜드명 표시
- **네비게이션 탭**: 우측 상단에 두 개의 탭 버튼
  - "주문하기" 탭: 주문하기 화면으로 이동하는 버튼
  - "관리자" 탭: 현재 화면 (선택된 상태로 표시)

#### 5.2.2 관리자 대시보드 요약 영역
- **섹션 제목**: "관리자 대시보드" 표시
- **주요 지표**: 요약 정보를 한 줄로 표시
  - "총 주문 N": 전체 주문 건수
  - "주문 접수 N": 접수된 주문 건수
  - "재고 완료 0": 재고 조정이 완료된 항목 수 (또는 관련 지표)

#### 5.2.3 재고 현황 영역
- **섹션 제목**: "재고 현황" 표시
- **상품별 재고 카드**: 각 상품마다 카드 형태로 표시
  - **상품명**: 상품명과 온도 옵션 표시 (예: "아메리카노(ICE)", "아메리카노(HOT)", "카페라테")
  - **현재 재고**: "N개" 형식으로 현재 재고 수량 표시
  - **재고 조정 버튼**: 각 상품 카드에 세 개의 버튼 제공
    - "+" 버튼: 재고 증가 (1개씩 증가)
    - 숫자 입력 버튼: 직접 수량 입력 (선택사항)
    - "-" 버튼: 재고 감소 (1개씩 감소)

#### 5.2.4 주문 현황 영역
- **섹션 제목**: "주문 현황" 표시
- **주문 목록**: 대기 중인 주문들을 시간순으로 표시
  - **주문 정보**: 각 주문 항목에 다음 정보 표시
    - 주문 일시: "N월 N일 HH:MM" 형식 (예: "1월 23일 13:00")
    - 주문 상품: 상품명과 수량 표시 (예: "아메리카노(ICE) X 1")
    - 주문 금액: "N,NNN원" 형식으로 표시 (예: "4,000원")
  - **주문 접수 버튼**: 각 주문 항목 옆에 "주문접수" 버튼 배치

### 5.3 기능 요구사항

#### 5.3.1 대시보드 요약 기능
- 실시간으로 전체 주문 건수 집계 및 표시
- 접수된 주문 건수 집계 및 표시
- 관련 지표를 자동으로 계산하여 표시

#### 5.3.2 재고 관리 기능
- 모든 상품의 현재 재고 수량을 실시간으로 조회 및 표시
- "+" 버튼 클릭 시 해당 상품의 재고를 1개씩 증가
- "-" 버튼 클릭 시 해당 상품의 재고를 1개씩 감소 (최소 0개까지)
- 재고 변경 시 즉시 데이터베이스에 반영
- 재고 변경 후 화면에 즉시 업데이트된 수량 표시

#### 5.3.3 주문 현황 조회 기능
- 대기 중인 주문 목록을 시간순으로 조회 및 표시
- 각 주문의 상세 정보(일시, 상품, 수량, 금액) 표시
- 새로운 주문이 들어오면 목록에 자동으로 추가 표시

#### 5.3.4 주문 접수 기능
- "주문접수" 버튼 클릭 시 해당 주문을 접수 상태로 변경
- 주문 접수 후 주문 상태를 "접수됨"으로 업데이트
- 접수된 주문은 주문 현황 목록에서 제거하거나 별도로 표시
- 주문 접수 시 대시보드의 "주문 접수" 지표 자동 업데이트

### 5.4 데이터 요구사항

#### 5.4.1 재고 데이터
- 상품 ID
- 상품명
- 온도 옵션 (ICE/HOT)
- 현재 재고 수량

#### 5.4.2 주문 데이터
- 주문 ID
- 주문 일시
- 주문 상태 (대기/접수됨/준비중/완료)
- 주문 항목 목록:
  - 상품 ID
  - 상품명
  - 수량
  - 옵션 정보
  - 개별 가격
- 총 주문 금액

#### 5.4.3 대시보드 지표 데이터
- 총 주문 건수 (전체 주문 수)
- 주문 접수 건수 (접수된 주문 수)
- 기타 관련 지표

### 5.5 사용자 인터랙션

#### 5.5.1 재고 조정 플로우
1. 관리자가 재고 현황 영역에서 상품 카드 확인
2. "+" 또는 "-" 버튼 클릭
3. 재고 수량이 즉시 업데이트되어 화면에 반영
4. 변경 사항이 데이터베이스에 저장됨

#### 5.5.2 주문 접수 플로우
1. 관리자가 주문 현황 영역에서 새로운 주문 확인
2. 주문 정보(일시, 상품, 수량, 금액) 검토
3. "주문접수" 버튼 클릭
4. 주문 상태가 "접수됨"으로 변경되고 목록에서 제거 또는 이동
5. 대시보드의 "주문 접수" 지표 자동 업데이트

### 5.6 UI/UX 요구사항
- 반응형 디자인으로 다양한 화면 크기에 대응
- 직관적인 버튼 배치와 명확한 라벨링
- 재고 변경 시 즉시 시각적 피드백 제공
- 주문 접수 시 명확한 상태 변경 표시
- 대시보드 지표는 실시간으로 업데이트
- 주문 목록은 최신 주문이 상단에 표시되도록 정렬
- 재고 수량은 명확하고 읽기 쉽게 표시
- 주문 정보는 한눈에 파악할 수 있도록 구조화하여 표시

## 6. 백엔드(PRD) - 데이터 모델 / 흐름 / API 설계

### 6.1 데이터 모델

#### 6.1.1 Menus (커피 메뉴)
- **필수 필드**
  - 메뉴 ID
  - 커피 이름
  - 설명
  - 가격
  - 이미지(URL 또는 경로)
  - 재고 수량

#### 6.1.2 Options (메뉴 옵션)
- **필수 필드**
  - 옵션 ID
  - 옵션 이름
  - 옵션 가격
  - 연결할 메뉴 ID (어떤 메뉴에 속한 옵션인지)

#### 6.1.3 Orders (주문)
- **필수 필드**
  - 주문 ID
  - 주문 일시
  - 주문 상태
  - 주문 내용 (메뉴, 수량, 옵션, 금액)

> 참고: `Orders.주문 내용`은 관계형 DB에서는 JSON 컬럼 또는 상세 테이블(예: OrderItems/OrderItemOptions)로 구현할 수 있습니다. (6.2 스키마 참고)

### 6.2 데이터 스키마(예시) 및 관계

#### 6.2.1 스키마(예: PostgreSQL)
- **menus**
  - id (PK)
  - name (커피 이름)
  - description
  - price
  - image_url
  - stock_quantity
  - created_at / updated_at

- **options**
  - id (PK)
  - menu_id (FK -> menus.id)
  - name (옵션 이름)
  - price (옵션 가격)
  - created_at / updated_at

- **orders**
  - id (PK)
  - ordered_at (주문 일시)
  - status (주문 상태)
  - total_amount (주문 총액)
  - created_at / updated_at

- **order_items** (주문 내용의 메뉴/수량/금액)
  - id (PK)
  - order_id (FK -> orders.id)
  - menu_id (FK -> menus.id)
  - quantity (수량)
  - unit_price (주문 시점 메뉴 가격)
  - line_amount (unit_price * quantity)

- **order_item_options** (주문 내용의 옵션/금액)
  - id (PK)
  - order_item_id (FK -> order_items.id)
  - option_id (FK -> options.id)
  - option_price (주문 시점 옵션 가격)

#### 6.2.2 주문 상태 정의
- **기본 상태**: `주문 접수`
- **상태 변경 흐름(관리자)**: `주문 접수` -> `제조 중` -> `완료`

### 6.3 데이터 스키마를 위한 사용자 흐름

#### 6.3.1 주문하기 화면: 메뉴 조회
- 백엔드는 `Menus` 데이터를 조회해 브라우저 화면에 표시할 수 있도록 제공한다.
- 이때 `Menus`의 **재고 수량(stock_quantity)**은 **관리자 화면**에서만 표시한다. (주문하기 화면에서는 노출하지 않음)

#### 6.3.2 주문하기 화면: 장바구니 담기
- 사용자가 앱 화면에서 커피 메뉴를 선택해 담는다.
- 선택 정보(메뉴, 수량, 옵션, 금액)는 프런트의 장바구니에 표시된다.

#### 6.3.3 주문하기: 주문 저장
- 장바구니에서 `주문하기` 버튼을 클릭하면 백엔드는 주문 정보를 `Orders`에 저장한다.
- `Orders`에는 주문 시간과 주문 내용(메뉴, 수량, 옵션, 금액)을 담는다.
- 주문 생성 시, 주문 수량만큼 `Menus.stock_quantity`를 감소시킨다.
  - 재고 차감은 **주문 생성과 같은 트랜잭션**으로 처리하여 데이터 정합성을 유지한다.
  - 재고가 부족한 경우 주문을 실패 처리한다.

#### 6.3.4 관리자 화면: 주문 현황 표시 및 상태 변경
- 백엔드는 `Orders`의 정보를 관리자 화면의 `주문 현황`에 표시하도록 제공한다.
- 주문의 기본 상태는 `주문 접수`이다.
- 관리자 화면에서 `주문 접수`를 클릭하면 `제조 중` -> `완료` 순으로 상태가 변경된다.

### 6.4 API 설계

#### 6.4.1 메뉴 목록 조회 (주문하기 화면)
- **목적**: `주문하기` 메뉴를 클릭하면 데이터베이스에서 커피 메뉴 목록을 불러와서 보여준다.
- **Endpoint**: `GET /api/menus`
- **응답(예시)**:
  - 메뉴 목록: id, name, description, price, image_url
  - 관리자용 조회가 필요하면 `GET /api/admin/menus`에서 stock_quantity 포함

#### 6.4.2 주문 생성 (주문하기 버튼)
- **목적**: 사용자가 커피를 선택하고 주문하기 버튼을 클릭하면 주문 정보를 데이터베이스에 저장한다.
- **Endpoint**: `POST /api/orders`
- **요청(예시)**:
  - items: [{ menu_id, quantity, option_ids[] }]
- **처리 규칙**
  - 주문 정보를 `orders / order_items / order_item_options`에 저장
  - 주문 정보에 따라 메뉴 목록의 재고(`menus.stock_quantity`)도 함께 수정(차감)
  - 기본 주문 상태는 `주문 접수`

#### 6.4.3 주문 조회 (주문 ID로 단건 조회)
- **목적**: 주문 ID를 전달하면 해당 주문 정보를 보여준다.
- **Endpoint**: `GET /api/orders/:orderId`
- **응답(예시)**:
  - order: id, ordered_at, status, items(메뉴/수량/옵션/금액), total_amount

#### 6.4.4 관리자: 주문 목록 조회 및 상태 변경
- **주문 목록 조회**
  - **Endpoint**: `GET /api/admin/orders`
  - **정렬**: 최신 주문이 상단(ordered_at DESC)
- **주문 상태 변경**
  - **Endpoint**: `PATCH /api/admin/orders/:orderId/status`
  - **처리**: `주문 접수` -> `제조 중` -> `완료` 순서로 상태 전환
